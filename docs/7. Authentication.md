# Authentication

> Stable and running in production across several projects. The API is close to final, with only small changes expected before 1.0.

Turn on auth with the [`auth` option](#optionsauth) and a `store`. It registers the login routes for your providers and loads the signed-in user onto [`ctx.user`](#ctxuser) on every request, so protecting a route is a one-line check:

```js
import server from '@server/next';
import kv from 'polystore';

export default server({
  store: kv(new Map()),
  auth: 'cookie:github',
})
  .get('/me', (ctx) => ctx.user || 401);
// open /auth/login/github to sign in
```

The `auth` value is `'<strategy>:<provider>'`. The strategy is `cookie` (a browser session cookie) or `token` (a bearer token for APIs), see [Strategies](#strategies). The provider is one of `email`, `github`, `google`, `microsoft`, `discord`, `facebook`, `apple`. The string form takes a single provider; for several, use the object form below with a `providers` array. For machine-to-machine access there's also a provider-less [`key`](#shared-secret-key) strategy that checks a single shared secret.

This guide covers:

- [Users and sessions](#users-and-sessions): the two stores auth derives from your `store`.
- [Configuring auth](#configuring-auth): the object form, with `redirect`, `cleanUser` and custom stores.
- [OAuth providers](#oauth-providers): GitHub, Google, Microsoft, Discord, Facebook and Apple.
- [Email and password](#email-and-password): register, login and password changes with no third party.
- [Protecting routes](#protecting-routes): guarding a route with a one-line `ctx.user` check.
- [Strategies](#strategies): sessions for browsers, bearer tokens for APIs.
- [Shared-secret key](#shared-secret-key): a single API key for machine-to-machine access, no users.
- [Routes](#routes): every route the `auth` option registers.

### Users and sessions

Auth uses two key-value stores, both derived from your `store` by default:

- **Users** are kept under the `user:` prefix (an OAuth user keyed by its provider id, an email user by its email).
- **Sessions** are kept under the `auth:` prefix and last a week. The cookie or bearer token is just a session id; the user is looked up from it on each request.

## Configuring auth

The string form covers most cases, but the object form configures the rest:

```js
export default server({
  auth: {
    strategy: 'cookie',
    providers: ['google', 'github'],
    redirect: '/dashboard',  // after a cookie login
    store: users,            // where users are kept
    session: sessions,       // where sessions are kept
    cleanUser: (user) => {
      // shape what is stored and put on ctx.user
      const { internalId, ...rest } = user;
      return rest;
    },
  },
});
```

- **`redirect`**: where a `cookie` login sends the browser. Defaults to `/user`.
- **`cleanUser`**: runs on the user before it is stored and before it reaches `ctx.user`. By default it strips `password`; override it to drop or rename fields.
- **`store` / `session`**: override the derived stores (otherwise the server's `store` is used for both).

## OAuth providers

Each provider reads its credentials from the environment:

| Provider | Env vars |
|----------|----------|
| `github` | `GITHUB_ID`, `GITHUB_SECRET` |
| `google` | `GOOGLE_ID`, `GOOGLE_SECRET` |
| `microsoft` | `MICROSOFT_ID`, `MICROSOFT_SECRET` |
| `discord` | `DISCORD_ID`, `DISCORD_SECRET` |
| `facebook` | `FACEBOOK_ID`, `FACEBOOK_SECRET` |
| `apple` | `APPLE_ID`, `APPLE_TEAM_ID`, `APPLE_KEY_ID`, `APPLE_PRIVATE_KEY` |

In the provider's console set the authorized redirect (callback) URL to `https://<your-host>/auth/callback/<provider>`. Then send users to `GET /auth/login/<provider>`; the server handles the callback, upserts the user, and (for `cookie`) redirects to your `redirect`. The flow is CSRF-protected with a short-lived `state` cookie. An OAuth `ctx.user` carries `{ id, name, email, picture }` plus `provider` and `strategy`.

> **Apple** signs its client secret as a short-lived ES256 JWT from your `.p8` key, with no extra dependencies. Provide the key contents as `APPLE_PRIVATE_KEY`, your Team ID as `APPLE_TEAM_ID`, the key id as `APPLE_KEY_ID`, and your Services ID as `APPLE_ID`. Its callback is a `POST`.

## Email and password

The `email` provider keeps users in your `store`, no third party involved:

- `POST /auth/register/email`, body `{ email, password, ...fields }`. The email must contain `@` and the password must be at least 8 characters. Extra fields are saved on the user; the password is hashed, never stored or returned in clear.
- `POST /auth/login/email`, body `{ email, password }`.
- `PUT /auth/password/email`, body `{ previous, updated }`, for a signed-in user to change their password.

```js
// register, and the session is active right away
await fetch('/auth/register/email', {
  method: 'POST',
  body: JSON.stringify({
    email: 'a@b.com',
    password: 'a-long-password',
    name: 'Francisco', // extra fields are stored
  }),
});
```

Register and login finish the session the same way as OAuth (a cookie or a token, per strategy). An email `ctx.user` carries `{ id, email, ...fields }` plus `provider` and `strategy`.

## Protecting routes

`ctx.user` is `undefined` when nobody is signed in, so a one-line middleware guards a route:

```js
const requireUser = (ctx) => {
  if (!ctx.user) return 401;
};

export default server({ store, auth: 'cookie:github' })
  .get('/public', () => 'anyone')
  .get('/account', requireUser,
    (ctx) => `Hello ${ctx.user.email}`);
```

You can type `ctx.user` with a second generic to `server()`; see [`ctx.user`](#ctxuser).

## Strategies

- **`cookie:<provider>`** (browsers): on success a signed, `HttpOnly` `authentication` cookie is set (`SameSite=Lax`, plus `Secure` in production) and the browser is redirected to `redirect`.
- **`token:<provider>`** (APIs, mobile): login responds with JSON that includes a `token` (an opaque id stored server-side); send it on later requests as `Authorization: Bearer <token>`.
- **`jwt:<provider>`** (stateless APIs): login responds with a `token` that is a signed **JWT** (HS256, signed with your [`secret`](#optionssecret)) carrying the session itself. No server-side session is stored, so it scales without a shared session store; the trade-off is you can't revoke an individual token before it expires (one week), and a change to `secret` invalidates all of them. Send it the same way, as `Authorization: Bearer <jwt>`.

```js
// token / jwt flow (identical for the client)
const { token } = await fetch('/auth/login/email', {
  method: 'POST',
  body: JSON.stringify({ email, password }),
}).then((r) => r.json());

await fetch('/account', {
  headers: { Authorization: `Bearer ${token}` },
});
```

> **`jwt` needs a stable `SECRET`.** JWTs are signed with your [`secret`](#optionssecret). If none is set, Server.js generates a random one per process, so every token is invalidated on restart and can't be verified by other instances (you'll also see a startup warning). Always set the `SECRET` environment variable when using `jwt`. Because there's no server-side record, a leaked JWT is valid until it expires (one week) and rotating `secret` is the only way to revoke all tokens at once, so prefer `cookie`/`token` if you need instant, per-session revocation.

An invalid or expired token/cookie responds with `401`, and a failed OAuth `state` check is `403`. A wrong email or password on the email provider (and a JWT that can't be verified) currently responds with `500`. `POST /auth/logout` ends the session for `cookie`/`token`; for stateless `jwt` there's nothing to revoke server-side, so the client just discards the token.

## Shared-secret key

For a service-to-service API where there are no users, only a trusted caller, use the `key` strategy. It checks one shared secret sent as `Authorization: Bearer <key>` against `AUTH_KEY` (or `auth.key`) in constant time. There are no providers, no store, no sessions and no login/logout routes.

Like every strategy, `key` only verifies the credential and sets [`ctx.user`](#ctxuser); it does **not** block unauthenticated requests on its own. Guard the routes you want to protect with a `ctx.user` check, exactly as in [Protecting routes](#protecting-routes):

```js
// AUTH_KEY=s3cr3t-api-key in the environment
const requireKey = (ctx) => {
  if (!ctx.user) return 401;
};

server({ auth: 'key' })
  .get('/data', requireKey, () => ({ ok: true }));
```

Pass the secret inline instead of the env var with the object form, `auth: { strategy: 'key', key: 'secret' }`. On a match `ctx.user` is a fixed marker, `{ id: 'key', strategy: 'key', provider: 'key' }` (there's no user record, so it carries no `email` or other fields); no header leaves the request anonymous (`ctx.user` undefined); a wrong or malformed header responds with `401`. Booting without a key configured throws, so a missing `AUTH_KEY` fails fast at startup rather than starting up unprotected.

## Routes

| Method | Route | Provider |
|--------|-------|----------|
| `GET` | `/auth/login/<provider>` | OAuth |
| `GET` | `/auth/callback/<provider>` | OAuth (Apple `POST`) |
| `POST` | `/auth/register/email` | email |
| `POST` | `/auth/login/email` | email |
| `PUT` | `/auth/password/email` | email |
| `POST` | `/auth/logout` | all except `key` |

The `key` strategy registers none of these; it only checks the `Authorization` header on your own routes.
